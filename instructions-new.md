THE DESIGN SYSTEM: I'm attaching a style reference (the Dia Browser system — "prism on white stationery"). Save it as DESIGN.md in the repo, adapt it for a product UI rather than a marketing site, and treat it as law:
- Canvas #F8F8F8, frosted cards rgba(255,255,255,0.9) with backdrop-blur(24px), the single rgba(0,0,0,0.08) 8px shadow, 30px card radius, nothing sharper than 10px anywhere. Replace whatever surface/elevation system exists today with this — one shadow, frosted layering, no hard borders.
- One typeface, three weights, nothing above 500. Featherweight (300) display type with tight negative tracking is the typographic signature. ABC Oracle isn't freely licensed, so substitute a close stand-in (e.g. DM Sans light) and note it in DESIGN.md. Strip out any bold-heavy hierarchy the current build has — hierarchy comes from size and weight contrast at the light end, not from 600/700.
- Near-achromatic everywhere. Audit the current palette and collapse it: any chromatic accents, colored buttons, or status colors get rationalized into the gray ramp (#000 / #636363 / #959595 / #D9D9D9) — except where color carries real meaning (errors, blast-radius warnings), where the gradient's --red and --blue tokens may appear as rare micro-accents.
- The spectrum gradient (pink → red → amber → lavender → blue) becomes the ONLY chromatic element and the signature of AI activity: when the agent is thinking, streaming, or materializing components on the canvas, the gradient is how the interface breathes — ambient glow, sweep animations, refraction. Never as text color or button fill. Buttons go neutral #D9D9D9 / ghost pills; kill any loud CTA colors in the current build.
- Where the Dia doc assumes marketing-page scale (72px heroes, 120px gaps), translate to app density: display weights for empty states and the gallery, tighter rhythm inside the working canvas. Document every adaptation in DESIGN.md so it stays the single source of truth.
- One deliberate departure from the doc: Dia's motion is smooth-curves-only, but I want this product to feel alive. Keep their 0.2s ease for micro-interactions and the theatrical cubic-bezier gradient sweeps, but layer in spring physics for gestures, the gallery zoom, and shared-element transitions. Liquid glass, bouncy, Apple-y. Write that override into DESIGN.md too.

TOOLING: We're on Tailwind — set up the DESIGN.md tokens as Tailwind theme variables so the system is enforced in code, not by convention. Bring in the Aceternity UI kit for the expressive layer (cards, spotlight/hover effects, animated backgrounds, text effects) where it beats what we have — but bend every Aceternity component to DESIGN.md before it ships; nothing lands in its default look. Where the existing custom components are already better than an Aceternity swap, keep and restyle them instead. You can go as far as custom WebGL/GLSL shaders and canvas effects where the kit can't reach — especially for component materialization, the gallery zoom, and token streaming.

SWEAT THE MOMENTS THAT DEFINE THE PRODUCT:
- Streaming and materialization: when the agent builds, it should feel like the app is being refracted into existence — gradient-driven, alive — not a loading state. If the current build uses spinners or plain typewriter streaming, that's first against the wall.
- The gallery: every app as a live frosted preview card floating on the canvas, gradient glowing softly beneath — premium portfolio, not a file list. The zoom between gallery and build view should be one continuous shared-element transition.
- Generated apps inherit the system: anything Sense Build produces should come out in the same frosted, restrained, art-directed language — every app from the same studio, no matter how lazy the prompt. If the current output templates don't match DESIGN.md, retheme them.
- The two-layer permission model and publish/share flows: keep them legible and calm — the org-wide blast-radius moment stays unmissable, but render it elegantly within the new system (this is one place a gradient --red micro-accent earns its keep).

Think through what the restyle changes about usability as you go — the new system is quieter than most builder UIs, so make sure state, hierarchy, and affordances still read clearly without color doing the work. Hide the machinery; surface the meaning.
